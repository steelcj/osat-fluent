# osat-fluent -- Archetype 5 -- Self-Contained Binary

Version: 0.1.0
Status: Draft
Style Guide: style-guide--versioned-documents-in-unrendered-markdown-v0-1-0

## Abstract

This document defines the pattern for building an OSAT Fluent tool that installs a self-contained binary distributed via GitHub Releases. It is the implementation guide for Archetype 5 (User XDG / Manual User Install) as defined in `software-installation-archetypes-v0-1-0` and the paths and permissions defined in `osat--user-space-installation-specification-v0-2-0`. A tool built to this pattern installs without elevation, lives entirely within the invoking user's home directory, verifies every download against a published checksum, works offline after first install via a local archive, and is fluent in the conventions of Linux, macOS, and Windows.

## 1. When to use this pattern

Use this pattern when all four of the following are true.

The tool is a self-contained binary with no runtime dependency. Go and Rust tools are the canonical examples. The binary runs directly without Python, Node, a JVM, or any other interpreter.

The tool is distributed via GitHub Releases. The installer queries the GitHub Releases API to detect the latest stable version and downloads the release asset directly. Tools distributed via other mechanisms require adaptation to the version detection and download sections only.

The tool is intended for a single user on the invoking machine. It installs to user-space paths under the home directory. No elevation is required or permitted.

The tool is a point-in-time binary, not a package with dependencies. If the tool requires other packages to be present at runtime, use the Archetype 6 pattern instead.

## 2. Repository layout

Every tool built to this pattern has the following repository structure:

```text
<tool-name>/
├── .gitignore
├── README.md
├── ROADMAP.md
├── create-repo-dirs.py
├── install-<tool-name>.py
└── scripts/
    ├── nix/
    │   └── <tool-name>
    └── windows/
        ├── <tool-name>.cmd
        └── <tool-name>.ps1
```

`create-repo-dirs.py` is generated by running `python3 en/utils/create-repo-dirs.py <tool-name>` from the osat-fluent repository. The versioned binary directories installed by the tool are excluded from git by the `.gitignore` pattern `[0-9]*.[0-9]*.[0-9]*/`, which the installer adds automatically on first run if not already present.

## 3. Install layout on disk

The installer writes to platform-native user-space paths only. It never touches system paths and refuses to run as root.

### 3.1 Linux and macOS

```text
~/.local/bin/<tool-name>                              wrapper
~/.local/share/<tool-name>/<version>/<binary>         versioned binary
~/.local/share/<tool-name>/archive/<version>/<binary> local archive
~/.config/<tool-name>/env                             env file (user-created)
~/.local/state/<tool-name>/                           state and logs
~/.private/<tool-name>/                               credentials
```

### 3.2 Windows

```text
%LOCALAPPDATA%\Programs\<tool-name>.cmd               cmd.exe wrapper
%LOCALAPPDATA%\Programs\<tool-name>.ps1               PowerShell wrapper
%LOCALAPPDATA%\<tool-name>\<version>\<binary>.exe     versioned binary
%LOCALAPPDATA%\<tool-name>\archive\<version>\         local archive
%APPDATA%\<tool-name>\env.ps1                         env file (user-created)
%LOCALAPPDATA%\<tool-name>\logs\                      state and logs
%USERPROFILE%\.private\<tool-name>\                   credentials
```

## 4. The installer

### 4.1 Structure

The installer is a single Python 3 script using only the standard library. It has no external dependencies beyond Python 3.8 and network access on first install.

```python
#!/usr/bin/env python3
"""
install-<tool-name>.py

One-paragraph description of what this installer does, the layout it
produces, the platforms it supports, and how to run it.

Usage: python3 install-<tool-name>.py
"""

from __future__ import annotations

import bz2          # if the tool uses bz2 archives
import hashlib
import json
import os
import platform
import shutil
import stat
import subprocess
import sys
import tarfile      # if the tool uses tar.gz archives
import tempfile
import urllib.error
import urllib.request
import zipfile      # if the tool uses zip archives
from pathlib import Path

API_URL      = "https://api.github.com/repos/<org>/<repo>/releases/latest"
RELEASE_BASE = "https://github.com/<org>/<repo>/releases/download"
USER_AGENT   = "<tool-name>-installer"
REPO_DIR     = Path(__file__).resolve().parent

ASSET_PLATFORMS = {
    ("Linux",   "x86_64"):  ("<linux-fragment>",   "<ext>", "<binary>"),
    ("Linux",   "aarch64"): ("<linux-arm-fragment>","<ext>", "<binary>"),
    ("Darwin",  "x86_64"):  ("<mac-fragment>",      "<ext>", "<binary>"),
    ("Darwin",  "arm64"):   ("<mac-arm-fragment>",  "<ext>", "<binary>"),
    ("Windows", "AMD64"):   ("<win-fragment>",       "zip",  "<binary>.exe"),
}
```

### 4.2 Platform paths

All path constants are derived from a single `platform_paths()` function called once in `main()`. No module-level path constants are used.

```python
def platform_paths() -> dict:
    home   = Path.home()
    system = platform.system()

    if system == "Windows":
        local   = Path(os.environ.get(
            "LOCALAPPDATA", home / "AppData" / "Local"))
        appdata = Path(os.environ.get(
            "APPDATA", home / "AppData" / "Roaming"))
        return {
            "tool_dir":     local / "<tool-name>",
            "wrapper_dir":  local / "Programs",
            "wrapper_cmd":  local / "Programs" / "<tool-name>.cmd",
            "wrapper_ps1":  local / "Programs" / "<tool-name>.ps1",
            "config_dir":   appdata / "<tool-name>",
            "state_dir":    local / "<tool-name>" / "logs",
            "private_dir":  home / ".private" / "<tool-name>",
            "archive_dir":  local / "<tool-name>" / "archive",
        }

    data    = Path(os.environ.get(
        "XDG_DATA_HOME",   home / ".local" / "share"))
    bin_dir = Path(os.environ.get(
        "XDG_BIN_HOME",    home / ".local" / "bin"))
    cfg     = Path(os.environ.get(
        "XDG_CONFIG_HOME", home / ".config"))
    state   = Path(os.environ.get(
        "XDG_STATE_HOME",  home / ".local" / "state"))
    return {
        "tool_dir":    data    / "<tool-name>",
        "wrapper_dir": bin_dir,
        "wrapper":     bin_dir / "<tool-name>",
        "config_dir":  cfg     / "<tool-name>",
        "state_dir":   state   / "<tool-name>",
        "private_dir": home    / ".private" / "<tool-name>",
        "archive_dir": data    / "<tool-name>" / "archive",
    }
```

### 4.3 Standard functions

Every installer in this pattern implements the following functions with these contracts. The implementations may differ by tool but the function names, signatures, and return types are consistent across the collection.

`log(message: str) -> None` — prints a prefixed status line to stdout.

`fail(message: str) -> None` — prints a prefixed error to stderr and exits with code 1. Never returns.

`platform_paths() -> dict` — returns platform-native resolved paths as described in section 4.2.

`detect_platform() -> tuple[str, str]` — returns `(platform.system(), platform.machine())`. Calls `fail()` if the combination is not in `ASSET_PLATFORMS`.

`fetch(url: str) -> bytes` — downloads a URL and returns the raw bytes. Calls `fail()` on any network or HTTP error.

`latest_version() -> str` — queries the GitHub Releases API and returns the bare version string (no leading `v`). Calls `fail()` on unexpected response structure.

`binary_ok(binary_path: Path, version: str) -> bool` — returns True if the binary at `binary_path` exists, is executable, and reports the expected version. Never raises.

`render_wrapper(version: str, paths: dict) -> str` — reads the wrapper template for the current platform, verifies the version placeholder is present, and returns the rendered content with the placeholder replaced by `version`.

`ensure_gitignore() -> None` — adds the versioned binary exclusion pattern to `.gitignore` if not already present.

`install_binary(...) -> None` — downloads the release asset, verifies the checksum, extracts the binary, writes it to the versioned path, sets `700` permissions, archives a copy, and calls `fail()` if the installed binary does not pass `binary_ok()`.

`install_wrapper(wrapper_content: str, wrapper_path: Path, version: str) -> None` — writes the rendered wrapper, sets `700` permissions, and calls `fail()` if the wrapper does not pass `binary_ok()`.

`post_install_notes(paths: dict) -> None` — warns if the wrapper directory is not in PATH, and warns if another installation of the tool is shadowing the wrapper.

`main() -> None` — root guard, platform detection, version detection, idempotency check, install, post-install notes.

### 4.4 Platform detection

```python
def detect_platform() -> tuple[str, str]:
    key = (platform.system(), platform.machine())
    if key not in ASSET_PLATFORMS:
        fail(
            f"platform {key[0]}/{key[1]} is not supported; "
            "see ROADMAP.md for planned platform bringup"
        )
    return key
```

Do not silently fall back to a guess. Unsupported platforms must fail loudly.

### 4.5 Version detection

```python
API_URL = "https://api.github.com/repos/<org>/<repo>/releases/latest"

def latest_version() -> str:
    log("querying GitHub for the latest stable release")
    try:
        release = json.loads(fetch(API_URL))
    except json.JSONDecodeError as error:
        fail(f"could not parse GitHub API response: {error}")
    tag = release.get("tag_name", "")
    if not tag.startswith("v") or len(tag) < 2:
        fail(f"unexpected tag_name in API response: {tag!r}")
    return tag[1:]
```

### 4.6 Checksum verification

Two variants exist in the collection. Use the one that matches the tool's release conventions. Do not skip verification.

**Variant A — standalone checksums file** (Hugo, restic, rclone):

```python
def expected_checksum(checksums_text: str, asset_name: str) -> str:
    for line in checksums_text.splitlines():
        parts = line.split()
        if len(parts) == 2 and parts[1] == asset_name:
            return parts[0]
    fail(f"{asset_name} not found in checksums file")
    raise AssertionError("unreachable")
```

**Variant B — GitHub API digest field** (marp-cli):

```python
def digest_from_release(release: dict, asset_name: str) -> str:
    for asset in release.get("assets", []):
        if asset.get("name") == asset_name:
            digest = asset.get("digest", "")
            if digest.startswith("sha256:"):
                return digest[len("sha256:"):]
    fail(f"no SHA-256 digest found for {asset_name} in GitHub API response")
    raise AssertionError("unreachable")
```

In both cases, verify immediately after download and before touching the filesystem:

```python
actual = hashlib.sha256(archive_bytes).hexdigest()
if actual != expected:
    fail(f"checksum verification failed for {asset_name}; aborting")
```

### 4.7 Archive formats

Three archive formats appear in the collection. Determine the correct format by inspecting the actual release assets before writing the installer — do not assume from the tool name or ecosystem.

**tar.gz** — used by marp-cli (Linux, macOS) and Hugo:

```python
with tarfile.open(archive_path, "r:gz") as tf:
    member = tf.getmember(binary_name)
    tf.extract(member, path=workdir, filter="data")
```

**bz2** — used by restic. A compressed single file, not a tarball:

```python
binary_bytes = bz2.decompress(archive_bytes)
extracted = workdir / binary_name
extracted.write_bytes(binary_bytes)
```

**zip** — used by marp-cli (Windows), rclone, Hugo (Windows):

```python
with zipfile.ZipFile(archive_path, "r") as zf:
    if binary_name not in zf.namelist():
        fail(f"'{binary_name}' not found in archive; found: {zf.namelist()}")
    zf.extract(binary_name, path=workdir)
```

### 4.8 Binary health check

```python
def binary_ok(binary_path: Path, version: str) -> bool:
    if not (binary_path.is_file() and os.access(binary_path, os.X_OK)):
        return False
    try:
        result = subprocess.run(
            [str(binary_path), "--version"],
            capture_output=True, text=True, timeout=30, check=False,
        )
    except OSError:
        return False
    return result.returncode == 0 and version in result.stdout
```

Adapt the version subcommand (`--version`, `version`, `-v`) and the version string search to match the tool's actual output. Verify before accepting the install.

### 4.9 Permissions

All paths created by the installer use owner-only permissions following the principle of least privilege defined in `osat--user-space-installation-specification-v0-2-0`.

```python
# Directories and executables
path.chmod(stat.S_IRWXU)                        # 700

# Files
path.chmod(stat.S_IRUSR | stat.S_IWUSR)         # 600
```

The installer verifies permissions on subsequent runs and fails explicitly if any path is found broader than specified.

### 4.10 Local archive

After a successful verified install, copy the binary to the local archive before placing it at the versioned path. The archive enables offline reinstall and rollback without a network request.

```python
archive_path = paths["archive_dir"] / version / binary_name
archive_path.parent.mkdir(parents=True, exist_ok=True)
shutil.copyfile(extracted_binary, archive_path)
archive_path.chmod(stat.S_IRWXU)
```

On subsequent runs, check the archive before going to the network:

```python
archived = paths["archive_dir"] / version / binary_name
if binary_ok(archived, version):
    log(f"restoring v{version} from local archive")
    # install from archive, skip download
```

### 4.11 Idempotency check

```python
wrapper_content = render_wrapper(version, paths)
binary_current  = binary_ok(binary_path, version)
wrapper_path    = paths["wrapper"]  # nix; use wrapper_cmd on Windows
wrapper_current = (
    wrapper_path.is_file()
    and wrapper_path.read_text(encoding="utf-8") == wrapper_content
)

if binary_current and wrapper_current:
    log(f"<tool-name> v{version} is already installed and active; nothing to do")
    return
```

### 4.12 Root guard

```python
if hasattr(os, "geteuid") and os.geteuid() == 0:
    fail(
        "this installer writes to the invoking user's home directory; "
        "do not run with sudo"
    )
```

## 5. Wrapper templates

### 5.1 Nix

```sh
#!/bin/sh
# <tool-name> — rendered by install-<tool-name>.py; do not edit by hand
_cfg="${XDG_CONFIG_HOME:-$HOME/.config}/<tool-name>/env"
[ -f "$_cfg" ] && . "$_cfg"
_bin="${XDG_DATA_HOME:-$HOME/.local/share}/<tool-name>/__TOOL_VERSION__/<binary>"
exec "$_bin" "$@"
```

### 5.2 Windows cmd

```batch
@echo off
rem <tool-name>.cmd — rendered by install-<tool-name>.py; do not edit by hand
set "_cfg=%APPDATA%\<tool-name>\env.cmd"
if exist "%_cfg%" call "%_cfg%"
set "_bin=%LOCALAPPDATA%\<tool-name>\__TOOL_VERSION__\<binary>.exe"
"%_bin%" %*
```

### 5.3 Windows PowerShell

```powershell
# <tool-name>.ps1 — rendered by install-<tool-name>.py; do not edit by hand
$_cfg = "$env:APPDATA\<tool-name>\env.ps1"
if (Test-Path $_cfg) { . $_cfg }
$_bin = "$env:LOCALAPPDATA\<tool-name>\__TOOL_VERSION__\<binary>.exe"
& $_bin @args
```

The placeholder `__TOOL_VERSION__` is replaced by the installer at render time. Name the placeholder after the tool, for example `__MARP_VERSION__` in marp-tool, to make the template self-describing and to fail loudly if the wrong template is used.

## 6. README structure

Every tool built to this pattern follows this README section order exactly:

```markdown
# <tool-name>

## Description
## Requirements
## Install
## Upgrade
## Rollback
## Layout
## See also
```

The Install section shows the clone-and-run sequence. The Upgrade section shows the pull-and-rerun sequence. The Rollback section explains the side-by-side versioned layout and how to point the wrapper at a previous version. The Layout section shows the full directory tree for the current platform. See also links to upstream documentation, the osat-fluent repository, and related tools in the collection.

## 7. ROADMAP.md

`ROADMAP.md` documents known gaps and their planned resolution. It is not a backlog. Every new tool at minimum documents Windows wrapper bringup if wrappers are not yet validated, and any architectures in `ASSET_PLATFORMS` that are mapped but not yet tested on hardware.

## 8. Creating a new tool — checklist

1. Run `python3 en/utils/create-repo-dirs.py <tool-name>` from the osat-fluent repository to scaffold the directory structure.
2. Inspect the upstream release page to determine the exact asset filenames, archive format, and checksum mechanism. Do not assume — verify.
3. Write `install-<tool-name>.py` following sections 4.1 through 4.12. Start from the most structurally similar existing tool in the collection.
4. Populate `ASSET_PLATFORMS` with only the platforms you have verified or intend to verify immediately. Add others to `ROADMAP.md`.
5. Write the nix wrapper template in `scripts/nix/<tool-name>` following section 5.1.
6. Write the Windows wrapper templates in `scripts/windows/` following sections 5.2 and 5.3.
7. Write `README.md` following section 6.
8. Write `ROADMAP.md` following section 7.
9. Run the installer. Verify the binary reports the expected version. Verify rerunning produces "nothing to do". Verify the wrapper content matches the rendered template. Verify permissions are `700` on directories and executables.
10. Commit and push. Reference `osat-fluent--archetype-5--self-contained-binary-v0-1-0` in the repository README under See also.

## 9. Decisions and rationale

### 9.1 Why stdlib only

The installer is the first thing a user runs in a fresh environment. It must have no prerequisites beyond Python 3.8 itself. Every external dependency is a failure mode before the tool is even installed. We considered `requests` for HTTP and `packaging` for version comparison. Both would simplify the code. We rejected both because they require a prior install step.

### 9.2 Why side-by-side versioning

Side-by-side versioning under `~/.local/share/<tool-name>/<version>/` allows rollback without a reinstall and lets the installer verify a new version is healthy before repointing the wrapper. We considered a single `current` symlink. We rejected it because symlinks behave differently on Windows and obscure what is actually installed.

### 9.3 Why a rendered wrapper rather than a symlink

A rendered wrapper can source the env file, can be read and understood by the user, and extends cleanly to Windows without platform-specific logic at call time. A symlink cannot source an env file and does not work uniformly across platforms.

### 9.4 Why archive-first resolution

A tool installed with a local archive does not depend on GitHub being reachable for reinstall or rollback. The archive is the resilience layer. It is not a cache, it is a permanent record of every verified install. The network is only required when a version is not yet archived locally.

### 9.5 Why platform_paths() rather than module-level constants

Module-level path constants cannot vary by platform without conditional logic scattered throughout the installer. A single `platform_paths()` function called once in `main()` resolves all paths for the current platform and passes them explicitly to every function that needs them. The platform difference is absorbed once, at the boundary, and is invisible everywhere else.
